# ADR-002: Гарантії доставки email-сповіщень

**Статус:** Прийнято

**Дата:** 2026-05-10

**Автор:** Іван Дутов

## Контекст

Система надсилає два типи email:

- confirmation email після створення підписки;
- release notification email після виявлення нового релізу GitHub repository.

Потрібно обрати механізм передачі email-повідомлень у Sender і явно зафіксувати, які гарантії доставки він дає. Для цього етапу важливо зберегти просту монолітну архітектуру без додаткової storage-моделі для email delivery.

## Розглянуті варіанти

### 1. In-memory channel

API Server і Scanner передають повідомлення Sender-у через channel у межах одного процесу.

- **Плюси:** найпростіший підхід; не потребує додаткових таблиць, міграцій або окремого сервісу; добре підходить для навчальної ітерації.
- **Мінуси:** повідомлення не переживають падіння процесу; немає durable acknowledgement; channel може переповнитися.

### 2. Зовнішня або персистентна черга

Email-повідомлення зберігаються поза пам'яттю процесу і обробляються Sender-ом пізніше.

- **Плюси:** повідомлення можуть переживати рестарт; можна реалізувати retry, статуси й аудит.
- **Мінуси:** додає storage-модель, locking protocol, scheduler повторних спроб і додаткові тести конкурентної обробки.

## Прийняте рішення

Прийнято використовувати **in-memory channel** і явно вважати email delivery **best-effort**.

Обране рішення не виконує durable at-least-once delivery. Повідомлення може бути втрачено в таких випадках:

- процес впав після постановки повідомлення в channel, але до успішної доставки;
- channel переповнений, а producer використовує non-blocking send;
- Resend повернув помилку для batch-а;
- Sender взяв batch в обробку, але процес завершився до успішної доставки.

Це рішення прийняте як компроміс для цього етапу: воно дає end-to-end flow підписки, підтвердження, polling GitHub і відправки email без додаткової storage-моделі.

## Наслідки

### Позитивні

- Delivery pipeline залишається простим.
- Немає додаткових таблиць, міграцій або locking protocol.
- Scanner і API Server не виконують HTTP-виклики до email-провайдера напряму.
- Sender може батчити повідомлення до ліміту Resend batch API.

### Негативні

- Email delivery є best-effort, а не durable at-least-once.
- Confirmation email і release notification email можуть бути втрачені.
- Після втрати release notification система може вже мати оновлений `last_seen_tag`, тому повторний scan не обов'язково відновить повідомлення.
- При переповненому channel producers можуть drop-нути email і тільки залогувати проблему.
