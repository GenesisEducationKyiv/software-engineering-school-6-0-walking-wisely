# ADR-002: Гарантії доставки email-сповіщень

**Статус:** Прийнято

**Дата:** 2026-05-10

**Автор:** Іван Дутов

## Контекст

Система має надсилати email-сповіщення про нові релізи підписникам. Для такого сценарію втрата повідомлення є небажаною: користувач очікує отримати сповіщення про реліз, на який він підписався.

Вимога **at-least-once delivery** означає, що повідомлення може бути доставлене повторно, але не має бути втрачено. Поточна архітектура використовує in-memory канал між Scanner і Sender. Якщо процес падає після постановки повідомлення в канал, але до успішної доставки email, це повідомлення може бути втрачено.

Потрібно визначити механізм доставки для поточного етапу розвитку системи та зафіксувати, які гарантії він дає. Цільова вимога з `system-design.md` описує at-least-once delivery для production-рівня, але це рішення має також враховувати операційну складність, монолітну архітектуру сервісу і поточний обсяг delivery pipeline.

Через обмеження часу поточна реалізація фокусується на end-to-end функціональності: підписка, підтвердження, polling GitHub, формування email і відправка через провайдера. Durable delivery свідомо винесено в наступну ітерацію, бо вона потребує схеми outbox, міграцій, locking protocol, retry scheduler і тестування конкурентної обробки.

У цьому ADR рішення стосується насамперед **release notification emails**, які створює Scanner після виявлення нового релізу. Confirmation emails у current-state надсилаються API Server-ом напряму через Email Provider Integration і також мають best-effort гарантії, але не проходять через in-memory release notification channel.

У цьому ADR терміни мають таке значення:

- **Current:** реалізація, яку потрібно встигнути зробити в межах поточної ітерації. Release notifications використовують in-memory channel і дають best-effort delivery. Confirmation emails надсилаються напряму через Email Provider Integration і теж залишаються best-effort.
- **Target:** production-oriented реалізація. Вона використовує persistent email outbox у PostgreSQL і дає at-least-once delivery для release notification emails. Confirmation emails можуть бути додані до того самого outbox окремим наступним рішенням.

## Розглянуті варіанти

### 1. In-memory черга

Scanner передає повідомлення Sender-у через in-memory канал у межах одного процесу.

- **Плюси:** найпростіша реалізація; нульова операційна складність; не потребує додаткових таблиць або сервісів.
- **Мінуси:** повідомлення втрачаються при падінні процесу; немає durable acknowledgement; складно гарантувати at-least-once без додаткової логіки повторного виявлення релізів.

### 2. Персистентна черга в базі даних

Scanner записує pending-сповіщення в таблицю бази даних. Sender читає pending-записи, надсилає email і позначає запис як доставлений або видаляє його після успіху.

- **Плюси:** повідомлення переживають рестарт процесу; можна реалізувати retry, статуси, аудит і backoff; достатньо надійно для production без окремого брокера.
- **Мінуси:** додає таблиці та міграції; збільшує навантаження на базу; потрібно обробляти конкуренцію між Sender-ами, блокування і повторні спроби.

### 3. Зовнішня черга

Для доставки використовується message broker: Redis Streams, RabbitMQ, SQS або аналогічний сервіс з acknowledgement.

- **Плюси:** найкраще підходить для масштабування; має готові механізми acknowledgement, retry і dead letter queue; відділяє Scanner від Sender-а.
- **Мінуси:** суттєво ускладнює інфраструктуру; потребує окремого сервісу, моніторингу і налаштування; може бути надмірним для поточного масштабу в 10 000 активних підписок.

## Прийняте рішення

Прийнято використовувати **in-memory чергу** між Scanner-ом, який створює release notification email-повідомлення, і Sender-ом, який надсилає їх через email-провайдера.

Це рішення аргументується так:

- система залишається простішою: не потрібні окрема таблиця черги, locking protocol, scheduler для delayed jobs або message broker;
- черга відділяє створення email-повідомлень від їх фактичної доставки;
- Sender може батчити повідомлення і контролювати пропускну здатність взаємодії з email-провайдером;
- тимчасове уповільнення email-провайдера не має блокувати основний сценарій сканування репозиторіїв;
- операційний стан черги можна контролювати через метрики глибини черги та кількості dropped/failed повідомлень.

Це рішення дає **best-effort delivery**, а не повну at-least-once гарантію. Воно приймається як обмежене current-state рішення для поточного етапу, де через брак часу простота і низька операційна складність важливіші за durable delivery.

Для відповідності target-state архітектурі заплановано перейти на **персистентну чергу в базі даних**. У цьому варіанті Scanner має записувати notification events у durable storage, а Sender має читати pending-повідомлення, надсилати email і змінювати статус лише після успішної доставки або після класифікованої фінальної помилки. Детальна специфікація наведена в `docs/email-outbox-spec.md`.

Зовнішня черга не обирається як наступний крок, бо PostgreSQL-достатній варіант покриває поточний масштаб без окремого брокера.

## Наслідки

### Позитивні

- Рішення зберігає просту архітектуру delivery pipeline.
- Немає додаткових таблиць, міграцій, locking protocol або окремого message broker-а.
- Сканування репозиторіїв не залежить напряму від latency email-провайдера.
- Sender може батчити повідомлення і зменшувати кількість HTTP-запитів до email-провайдера.
- Метрики глибини черги та помилок доставки дають базову видимість стану pipeline.
- Контрольований shutdown може зменшити ризик втрати повідомлень, які вже перебувають у черзі.

### Негативні

- Повідомлення можуть бути втрачені при падінні процесу, переповненні черги або помилці email-провайдера після взяття повідомлення в обробку.
- Рішення не має durable acknowledgement і не виконує повну at-least-once гарантію.
- Якщо стан "реліз уже оброблено" оновлюється окремо від факту durable створення notification event, частина повідомлень може не бути відновлена повторним скануванням.
- Confirmation email також може бути втрачений, якщо API Server не зміг успішно доставити його через Email Provider Integration. У current-state він не захищений durable outbox-ом.
- Для відповідності `system-design.md` по release notification emails потрібна міграція на персистентну чергу або transactional outbox.

## План розвитку

Наступний production-oriented крок — замінити in-memory delivery queue для release notification emails на персистентну чергу в PostgreSQL:

- додати таблицю `email_outbox`, як описано в `docs/email-outbox-spec.md`;
- записувати повідомлення в одній транзакції з бізнес-подією, яка його створює;
- резервувати pending-повідомлення через `FOR UPDATE SKIP LOCKED`;
- зберігати `attempt_count`, `next_attempt_at`, `status` і останню помилку;
- оновлювати стан обробленого релізу лише після durable створення notification events, а не після best-effort enqueue;
- додати retry/backoff і dead-letter статус для повідомлень, які не вдалося доставити.

Confirmation emails можна перевести на той самий outbox пізніше, якщо потрібно гарантувати durable retry і для double opt-in flow. Це не блокує перший outbox-крок, бо основна target-вимога at-least-once стосується release notification emails.
