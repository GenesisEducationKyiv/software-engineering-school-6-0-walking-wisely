# ADR-001: Стратегія опитування GitHub API

**Статус:** Прийнято

**Дата:** 2026-05-10

**Автор:** Іван Дутов

## Контекст

Система відстежує релізи GitHub-репозиторіїв і надсилає email-сповіщення підписникам. GitHub REST API має обмеження на кількість запитів: 60 запитів на годину без автентифікації та 5000 запитів на годину з токеном.

Якщо Scanner робить окремий запит для кожної підписки, то кількість запитів зростає разом із кількістю користувачів, а не разом із кількістю унікальних репозиторіїв. Наприклад, 1000 підписок на 100 унікальних репозиторіїв створюють 1000 запитів за цикл замість 100.

Потрібно зменшити кількість зайвих GitHub API запитів без додавання окремого scheduler-а, webhook lifecycle або додаткової таблиці стану репозиторіїв.

## Розглянуті варіанти

### 1. Перевірка кожної підписки окремо

- **Плюси:** найпростіша модель для розуміння.
- **Мінуси:** кількість GitHub API запитів росте разом із кількістю підписок і швидко вичерпує rate limit.

### 2. Дедуплікація per-repo

Scanner групує підписки за репозиторієм і робить один HTTP-запит на кожен унікальний репозиторій за цикл.

- **Плюси:** суттєво зменшує кількість запитів; не потребує додаткової інфраструктури; добре відповідає схемі `subscriptions`.
- **Мінуси:** система все ще залежить від polling; затримка виявлення релізу залежить від інтервалу сканування; запити все одно рахуються в GitHub rate limit.

### 3. Кешування latest release в Redis

GitHub client кешує відповідь latest release на короткий TTL.

- **Плюси:** зменшує кількість реальних HTTP-запитів, якщо той самий repo перевіряється або валідовується повторно в межах TTL.
- **Мінуси:** кеш може приховати новий реліз до завершення TTL; це не є точною гарантією виявлення релізу.

## Прийняте рішення

Прийнято використовувати **fixed-interval per-repo polling з Redis TTL-кешем**.

Scanner:

- запускається з фіксованим `SCANNER_INTERVAL`;
- читає унікальні підтверджені репозиторії через `SELECT DISTINCT repo`;
- викликає GitHub latest release один раз на repo за scan cycle;
- отримує підтверджених підписників repo;
- ставить email-повідомлення в in-memory channel, якщо `last_seen_tag` відрізняється від latest release tag;
- після успішної постановки хоча б одного повідомлення оновлює `last_seen_tag` для підтверджених підписок repo.

GitHub client:

- використовує `GITHUB_TOKEN`, якщо він налаштований;
- кешує latest release у Redis на 10 хвилин;
- повертає `ErrRepoNotFound` для `404`;
- повертає `RateLimitError` для `403` або `429`, включно з `Retry-After` або `X-RateLimit-Reset`, якщо вони доступні.

При GitHub rate limit або іншій помилці Scanner має логувати проблему і пропускати конкретний repo в цьому циклі. Окремий scheduler повторних спроб не вводиться; наступна спроба відбудеться в одному з наступних scan cycles.

## Обмеження рішення

- Немає conditional requests через `ETag` або `Last-Modified`.
- Немає збереженого `next_check_at`, `last_checked_at` або activity tier.
- Немає rate-budget aware scheduler-а, quota buffer або пріоритезації repo при дефіциті GitHub API budget.
- Redis TTL-кеш може додати stale delay до виявлення нового релізу.
- Якщо кількість унікальних repo значно зросте, fixed-interval polling може вичерпати GitHub API budget.

## Наслідки

### Позитивні

- Кількість запитів до GitHub API залежить від кількості унікальних репозиторіїв, а не від кількості підписників.
- Рішення просте і не потребує додаткових таблиць або окремого планувальника.
- Optional `GITHUB_TOKEN` дозволяє підняти GitHub API limit з 60 до 5000 запитів на годину.
- Redis cache зменшує дубльовані звернення до GitHub у межах короткого TTL.

### Негативні

- Система залишається polling-based, тому релізи не доставляються миттєво.
- Обрана поведінка не дає повного контролю над GitHub API budget.
- При rate limit repo просто пропускається до наступного scan cycle.
- Рішення не гарантує однакову затримку виявлення релізів при рості кількості унікальних repo.
