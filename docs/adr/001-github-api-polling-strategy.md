# ADR-001: Стратегія опитування GitHub API

**Статус:** Прийнято

**Дата:** 2026-05-10

**Автор:** Іван Дутов

## Контекст

Система відстежує релізи GitHub-репозиторіїв і надсилає email-сповіщення підписникам. GitHub REST API має обмеження на кількість запитів: 60 запитів на годину без автентифікації та 5000 запитів на годину з токеном.

Якщо Scanner робить окремий запит для кожної підписки, то кількість запитів зростає разом із кількістю користувачів, а не разом із кількістю унікальних репозиторіїв. Наприклад, 1000 підписок на 100 унікальних репозиторіїв створюють 1000 запитів за цикл замість 100. При зростанні навантаження це швидко вичерпує rate limit і робить сервіс нестабільним.

Поточна реалізація вже робить базову **дедуплікацію per-repo**: Scanner з фіксованим `SCANNER_INTERVAL` отримує `SELECT DISTINCT repo` для підтверджених підписок і перевіряє кожен унікальний репозиторій один раз за цикл. GitHub client використовує optional `GITHUB_TOKEN`, повертає rate-limit помилки через `RateLimitError` і має Redis TTL-кеш latest release на 10 хвилин.

Водночас поточна реалізація не є повною production-стратегією GitHub rate-limit handling. Вона не зберігає `ETag` / `Last-Modified`, не виконує conditional requests, не читає й не планує роботу за `x-ratelimit-remaining` / `x-ratelimit-reset`, не має `next_check_at`, activity tiers або quota buffer. При rate-limit помилці Scanner лише логує проблему і пропускає конкретний репозиторій у поточному циклі.

Потрібно обрати стратегію опитування GitHub API, яка:

- зменшує кількість зайвих запитів;
- витримує цільове навантаження у 10 000 активних підписок і приблизно 500-700 унікальних репозиторіїв;
- дозволяє масштабувати кількість підписок;
- адаптує частоту polling до реальної активності репозиторіїв;
- явно керує GitHub API rate budget і не покладається на неявні припущення про ліміти.

Через обмеження часу current-state реалізація залишається простішою: fixed-interval scan, per-repo дедуплікація, optional authenticated requests і короткий Redis cache. Target-state має перейти на activity-tier scheduler з conditional requests і явним rate budget management.

## Розглянуті варіанти

### 1. Дедуплікація per-repo

Scanner групує підписки за репозиторієм і робить один HTTP-запит на кожен унікальний репозиторій незалежно від кількості підписників.

- **Плюси:** проста зміна логіки Scanner-а; суттєво зменшує кількість запитів; не потребує додаткової інфраструктури; є обов'язковою базою для цільового навантаження.
- **Мінуси:** система все ще залежить від polling; затримка виявлення релізу залежить від інтервалу сканування; запити все одно рахуються в rate limit.

### 2. GitHub Webhooks

Система підписується на webhook `release` від GitHub і отримує події напряму замість періодичного polling.

- **Плюси:** майже миттєве отримання подій; мінімальне використання GitHub API; краща масштабованість для великої кількості репозиторіїв.
- **Мінуси:** потрібен публічний URL; потрібен адміністраторський доступ до репозиторію; потрібно керувати реєстрацією webhook-ів; складніше обробляти повторні доставки, підписи та недоступність endpoint-а.

### 3. Conditional requests через ETag / If-Modified-Since

Scanner використовує HTTP-кешування GitHub API. Для повторних перевірок надсилаються заголовки `ETag` або `If-Modified-Since`, а відповідь `304 Not Modified` не витрачає основний rate limit GitHub API.

- **Плюси:** зменшує споживання primary rate limit для репозиторіїв без змін; не змінює архітектуру доставки подій; добре підходить для release polling, бо релізи змінюються значно рідше, ніж виконується перевірка.
- **Мінуси:** не зменшує кількість HTTP-викликів; не допомагає, якщо відповідь змінилася і повертається `200 OK`; потребує авторизованих запитів і зберігання cache metadata; не є самостійною стратегією масштабування.

### 4. Rate-budget aware polling

Scanner планує перевірки з урахуванням GitHub API rate limit: читає `x-ratelimit-*` headers, контролює кількість запитів у вікні, обмежує concurrency і відкладає частину перевірок при наближенні до ліміту.

- **Плюси:** явно захищає систему від перевищення rate limit; робить поведінку передбачуваною при 500-700 унікальних репозиторіях; дає операційні метрики для алертингу.
- **Мінуси:** не усуває сам GitHub limit; при дефіциті rate budget частина репозиторіїв може перевірятися пізніше; для жорсткої 10-хвилинної SLA може знадобитися вищий rate limit або webhook-підхід.

### 5. Adaptive activity-tier polling

Scanner розподіляє репозиторії за activity tiers на основі часу від останнього знайденого релізу і планує наступну перевірку з різним інтервалом:

| Tier | Умова                                                               | Polling interval |
| ---- | ------------------------------------------------------------------- | ---------------- |
| Hot  | реліз був протягом останніх 7 днів або стан активності ще невідомий | 10 хвилин        |
| Warm | релізів не було >= 7 днів                                           | 1 година         |
| Cold | релізів не було >= 30 днів                                          | 24 години        |

Новий репозиторій або репозиторій без збереженого release state починає з `Hot`, щоб перша перевірка не відкладалася. Після виявлення нового релізу репозиторій повертається в `Hot`. Якщо релізів немає протягом відповідного періоду, репозиторій поступово переходить у `Warm` і `Cold`.

- **Плюси:** різко зменшує кількість HTTP-викликів для неактивних репозиторіїв; зменшує ризик несподіваного споживання token budget; дозволяє обслуговувати більше унікальних репозиторіїв без зміни архітектури.
- **Мінуси:** 10-хвилинна затримка виявлення більше не гарантується для всіх репозиторіїв; для `Warm` репозиторіїв очікувана затримка може бути до 1 години, для `Cold` - до 24 годин; потрібно зберігати `last_release_published_at`, `last_checked_at`, `next_check_at` і поточний tier.

### 6. GitHub App з installation access token

Система реєструється як GitHub App і використовує installation access token для запитів до GitHub API. Для GitHub App installation мінімальний primary rate limit становить 5 000 запитів на годину. Для інсталяцій не в GitHub Enterprise Cloud ліміт масштабується: після 20 репозиторіїв додається 50 запитів на годину за кожен репозиторій, а після 20 користувачів в організації додається 50 запитів на годину за кожного користувача. Максимальний ліміт для такої інсталяції становить 12 500 запитів на годину. Для інсталяції в GitHub Enterprise Cloud organization ліміт становить 15 000 запитів на годину.

- **Плюси:** дає більший GitHub API budget для репозиторіїв або організацій, де App можна встановити; це кращий варіант для системної інтеграції, ніж персональний токен; модель GitHub App дає контрольовані permissions і окремі installation tokens; підходить як шлях масштабування за межі поточного профілю у 500-700 унікальних репозиторіїв.
- **Мінуси:** GitHub App потрібно встановити на репозиторії або організації, до яких вона має доступ; це не розв'язує сценарій довільного публічного `owner/repo` без участі власника репозиторію; secondary rate limits все одно діють; потрібна додаткова інфраструктура для installation lifecycle, token refresh і маршрутизації запитів по інсталяціях.

## Прийняте рішення

Для поточної реалізації прийнято **fixed-interval per-repo polling з Redis TTL-кешем** як навчальний/current-state компроміс.

Current-state Scanner групує підтверджені підписки за унікальним `repo`, перевіряє кожен репозиторій у межах фіксованого `SCANNER_INTERVAL` і створює notification email для підписників, якщо latest release tag відрізняється від `last_seen_tag`. Це вже прибирає найгірший сценарій per-subscription polling, але не дає повного контролю над GitHub API budget.

Redis TTL-кеш latest release у current-state зменшує кількість реальних HTTP-викликів, якщо той самий repo перевіряється або валідовується повторно в межах TTL. Але це не є цільовою release detection guarantee: TTL-кеш може приховати новий реліз до завершення TTL і не дає тих самих властивостей, що conditional request з `ETag` / `If-None-Match`.

Для target-state production-архітектури обрано комбінацію **дедуплікації per-repo**, **adaptive activity-tier polling**, **conditional requests** і **rate-budget aware scheduler**.

Scanner має групувати активні підписки за унікальним репозиторієм, виконувати один запит до GitHub API на репозиторій, а після виявлення нового релізу створювати сповіщення для всіх підписників цього репозиторію.

Scanner не має сканувати всі репозиторії з однаковим 10-хвилинним інтервалом. Він має зберігати для кожного репозиторію `last_seen_release`, `last_release_published_at`, `last_checked_at`, `next_check_at`, `etag`, `last_modified` і activity tier. Планувальник вибирає до перевірки тільки репозиторії з `next_check_at <= now()`.

Базові інтервали:

- `Hot`: 10 хвилин для нових репозиторіїв, репозиторіїв з невідомим станом і репозиторіїв, де останній реліз був менше 7 днів тому.
- `Warm`: 1 година, якщо останній реліз був 7 або більше днів тому.
- `Cold`: 24 години, якщо останній реліз був 30 або більше днів тому.

Після `200 OK` з новим релізом Scanner створює notification events, оновлює release state, переводить репозиторій у `Hot` і планує наступну перевірку через 10 хвилин. Після `304 Not Modified` або `200 OK` без нового релізу Scanner перераховує tier на основі `last_release_published_at` і виставляє наступний `next_check_at` згідно з tier interval. Для невідомого `last_release_published_at` репозиторій залишається `Hot`, поки Scanner не отримає достовірний release state або не зафіксує, що latest release відсутній.

Scanner має бути **rate-budget aware**: використовувати authenticated requests, контролювати `x-ratelimit-remaining` / `x-ratelimit-reset`, обмежувати concurrency і не продовжувати polling, якщо це веде до rate-limit помилок. Планувальник має резервувати частину primary quota для subscribe-time repository validation, ручних повторів і operational margin. Для стандартного authenticated limit 5 000 req/год Scanner може використовувати не більше 90% quota на background polling, тобто приблизно 4 500 primary-counting requests/год, залишаючи 10% buffer.

ETag не вважається capacity guarantee. Якщо conditional request повертає `304 Not Modified` і запит був коректно авторизований, він не витрачає primary rate limit. Але `200 OK` при новому релізі, зміні відповіді або втраті cache metadata витрачає quota. Тому capacity planning має рахувати worst-case primary-counting requests, а не припускати, що всі незмінені репозиторії будуть безкоштовними.

**Conditional requests через ETag / If-Modified-Since** прийнято використовувати як оптимізацію polling, а не як окрему архітектурну гарантію. Вони корисні для типового release polling, де більшість перевірок не знаходить нового релізу і отримує `304 Not Modified`. Але вони не повинні бути єдиним аргументом, що система витримає навантаження.

Для release polling не використовується TTL-кеш повних відповідей GitHub API як джерело істини, бо stale cache може приховати новий реліз і порушити tier-specific polling objective. Scanner має зберігати cache metadata (`ETag`, `Last-Modified`, rate-limit headers) разом зі станом репозиторію і виконувати conditional request до GitHub при кожній плановій перевірці. Короткий cache для subscribe-time repository validation допустимий, але він не замінює polling.

При оцінці з `system-design.md` приблизно 500-700 унікальних репозиторіїв з інтервалом 10 хвилин дають близько 3 000-4 200 HTTP-викликів на годину. Це вкладається у стандартний authenticated primary limit 5 000 запитів на годину на один token тільки з помірним запасом і тільки поки кількість унікальних репозиторіїв не росте. Adaptive intervals зменшують базове навантаження: наприклад, 100 `Hot`, 300 `Warm` і 300 `Cold` репозиторіїв потребують приблизно `100 * 6 + 300 * 1 + 300 / 24 = ~913` scheduled checks/год замість `700 * 6 = 4 200` checks/год.

10-хвилинна затримка виявлення є цільовою тільки для `Hot` репозиторіїв. Для `Warm` і `Cold` репозиторіїв система свідомо обмінює latency на стабільність rate budget: затримка може становити до 1 години або до 24 годин відповідно. Якщо бізнес-вимога вимагає 10-хвилинного detection bound для всіх репозиторіїв незалежно від активності, adaptive tiering не можна використовувати як основну стратегію; тоді потрібно збільшувати API budget або переходити на webhooks/GitHub App для відповідних репозиторіїв.

Якщо кількість унікальних репозиторіїв зростає за межі цього профілю і система перестає вкладатися в rate budget, потрібно змінювати одну з базових умов: збільшувати доступний GitHub API budget, зменшувати частоту polling для частини репозиторіїв, або переходити на webhook-модель для репозиторіїв, де доступні права адміністрування.

Якщо доступний budget у поточному rate-limit window недостатній навіть для всіх due checks, Scheduler має:

1. зберегти 10% quota buffer;
2. першими виконувати `Hot` репозиторії, потім `Warm`, потім `Cold`;
3. всередині tier сортувати за найстарішим `next_check_at`, щоб не створювати starvation;
4. відкладати решту перевірок до `x-ratelimit-reset` або до моменту, коли budget знову дозволить polling;
5. при `403` або `429` з primary rate limit чекати `x-ratelimit-reset`, а при secondary rate limit використовувати `Retry-After` або exponential backoff.

Отже, різниця між current і target така:

| Аспект                       | Current-state                                                 | Target-state                                                          |
| ---------------------------- | ------------------------------------------------------------- | --------------------------------------------------------------------- |
| Групування запитів           | Один scan на унікальний repo за цикл                          | Один scheduled check на унікальний repo                               |
| Інтервал                     | Фіксований `SCANNER_INTERVAL` для всіх repo                   | `Hot` / `Warm` / `Cold` activity tiers                                |
| Cache                        | Redis TTL-кеш latest release                                  | Збережені `ETag`, `Last-Modified` і conditional requests              |
| Rate budget                  | Optional token і реакція на `403` / `429` постфактум          | Планування за `x-ratelimit-*`, quota buffer і пріоритезація           |
| Scheduling state             | Немає `next_check_at`; scan бере всі confirmed repos          | `next_check_at`, `last_checked_at`, `last_release_published_at`, tier |
| Поведінка при дефіциті quota | Repo пропускається в поточному циклі після rate-limit помилки | Due checks відкладаються контрольовано до reset/backoff               |
| SLA detection                | Фіксований interval, але TTL-кеш може додати stale delay      | Один interval поточного tier; 10 хвилин тільки для `Hot`              |

GitHub Webhooks не обираються як базове рішення, бо вони вимагають публічного endpoint-а та прав адміністрування репозиторіїв, що погано сумісно з моделлю, де користувач підписується на довільний `owner/repo`.

GitHub App не використовується в поточному рішенні, але приймається як запланований майбутній напрям розвитку. Це хороший вибір для репозиторіїв або організацій, де користувачі готові встановити App, бо installation access token може дати до 12 500 запитів на годину, а для GitHub Enterprise Cloud organization - 15 000 запитів на годину. Такі ліміти потрібні для подальшого росту кількості унікальних репозиторіїв, але не є обов'язковими для поточного профілю навантаження.

## Наслідки

### Позитивні

- Кількість запитів до GitHub API залежить від кількості унікальних репозиторіїв, а не від кількості підписників.
- Неактивні репозиторії перевіряються рідше, тому система не витрачає однаковий budget на активні й давно неактивні репозиторії.
- 10% GitHub API quota залишається зарезервованою для subscribe-time validation, ручних операцій, retry та непередбачених змін у відповідях GitHub.
- Архітектура не потребує публічного webhook endpoint-а для довільних репозиторіїв.
- Логіка добре масштабується для сценаріїв, де багато користувачів підписані на однакові репозиторії.
- Rate-budget aware scheduler робить поведінку Scanner-а контрольованою при наближенні до GitHub API limit.
- Conditional requests зменшують primary rate limit usage для репозиторіїв без змін.
- GitHub App є зрозумілим шляхом збільшення API budget для репозиторіїв, де можлива інсталяція App.

### Негативні

- Система залишається polling-based, тому релізи не доставляються миттєво.
- Потрібно зберігати стан останнього побаченого релізу для кожного репозиторію.
- Потрібно зберігати й оновлювати scheduling metadata (`last_checked_at`, `next_check_at`, activity tier, cache metadata).
- 10-хвилинна затримка виявлення більше не є універсальною гарантією для всіх репозиторіїв; вона застосовується до `Hot` tier.
- Conditional requests не зменшують кількість HTTP-викликів і не гарантують економію rate limit при `200 OK` відповідях.
- Якщо кількість унікальних репозиторіїв суттєво перевищить 700, 10-хвилинний polling може вичерпати стандартний GitHub API limit без додаткового API budget або webhook-стратегії.
- Якщо rate budget недостатній, доведеться переглядати SLA polling, спосіб автентифікації або webhook-стратегію.
- GitHub App не покриває довільні репозиторії автоматично: для доступу через installation token потрібна встановлена App.
