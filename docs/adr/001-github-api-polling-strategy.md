# ADR-001: Стратегія опитування GitHub API

**Статус:** Прийнято

**Дата:** 2026-05-10

**Автор:** Іван Дутов

## Контекст

Система відстежує релізи GitHub-репозиторіїв і надсилає email-сповіщення підписникам. GitHub REST API має обмеження на кількість запитів: 60 запитів на годину без автентифікації та 5000 запитів на годину з токеном.

Якщо Scanner робить окремий запит для кожної підписки, то кількість запитів зростає разом із кількістю користувачів, а не разом із кількістю унікальних репозиторіїв. Наприклад, 1000 підписок на 100 унікальних репозиторіїв створюють 1000 запитів за цикл замість 100. При зростанні навантаження це швидко вичерпує rate limit і робить сервіс нестабільним.

Потрібно обрати стратегію опитування GitHub API, яка:

- зменшує кількість зайвих запитів;
- витримує цільове навантаження у 10 000 активних підписок і приблизно 500-700 унікальних репозиторіїв;
- дозволяє масштабувати кількість підписок;
- явно керує GitHub API rate budget і не покладається на неявні припущення про ліміти.

## Розглянуті варіанти

### 1. Дедуплікація per-repo

Scanner групує підписки за репозиторієм і робить один HTTP-запит на кожен унікальний репозиторій незалежно від кількості підписників.

- **Плюси:** проста зміна логіки Scanner-а; суттєво зменшує кількість запитів; не потребує додаткової інфраструктури; є обов'язковою базою для цільового навантаження.
- **Мінуси:** система все ще залежить від polling; затримка виявлення релізу залежить від інтервалу сканування; запити все одно рахуються в rate limit.

### 2. GitHub Webhooks

Система підписується на webhook `release` від GitHub і отримує події напряму замість періодичного polling.

- **Плюси:** майже миттєве отримання подій; мінімальне використання GitHub API; краща масштабованість для великої кількості репозиторіїв.
- **Мінуси:** потрібен публічний URL; потрібен адміністраторський доступ до репозиторію; потрібно керувати реєстрацією webhook-ів; складніше обробляти повторні доставки, підписи та недоступність endpoint-а.

### 3. Conditional requests через ETag / If-Modified-Since

Scanner використовує HTTP-кешування GitHub API. Для повторних перевірок надсилаються заголовки `ETag` або `If-Modified-Since`, а відповідь `304 Not Modified` не витрачає основний rate limit GitHub API.

- **Плюси:** зменшує споживання primary rate limit для репозиторіїв без змін; не змінює архітектуру доставки подій; добре підходить для release polling, бо релізи змінюються значно рідше, ніж виконується перевірка.
- **Мінуси:** не зменшує кількість HTTP-викликів; не допомагає, якщо відповідь змінилася і повертається `200 OK`; потребує авторизованих запитів і зберігання cache metadata; не є самостійною стратегією масштабування.

### 4. Rate-budget aware polling

Scanner планує перевірки з урахуванням GitHub API rate limit: читає `x-ratelimit-*` headers, контролює кількість запитів у вікні, обмежує concurrency і відкладає частину перевірок при наближенні до ліміту.

- **Плюси:** явно захищає систему від перевищення rate limit; робить поведінку передбачуваною при 500-700 унікальних репозиторіях; дає операційні метрики для алертингу.
- **Мінуси:** не усуває сам GitHub limit; при дефіциті rate budget частина репозиторіїв може перевірятися пізніше; для жорсткої 10-хвилинної SLA може знадобитися вищий rate limit або webhook-підхід.

### 5. GitHub App з installation access token

Система реєструється як GitHub App і використовує installation access token для запитів до GitHub API. Для GitHub App installation мінімальний primary rate limit становить 5 000 запитів на годину. Для інсталяцій не в GitHub Enterprise Cloud ліміт масштабується: після 20 репозиторіїв додається 50 запитів на годину за кожен репозиторій, а після 20 користувачів в організації додається 50 запитів на годину за кожного користувача. Максимальний ліміт для такої інсталяції становить 12 500 запитів на годину. Для інсталяції в GitHub Enterprise Cloud organization ліміт становить 15 000 запитів на годину.

- **Плюси:** дає більший GitHub API budget для репозиторіїв або організацій, де App можна встановити; це кращий варіант для системної інтеграції, ніж персональний токен; модель GitHub App дає контрольовані permissions і окремі installation tokens; підходить як шлях масштабування за межі поточного профілю у 500-700 унікальних репозиторіїв.
- **Мінуси:** GitHub App потрібно встановити на репозиторії або організації, до яких вона має доступ; це не розв'язує сценарій довільного публічного `owner/repo` без участі власника репозиторію; secondary rate limits все одно діють; потрібна додаткова інфраструктура для installation lifecycle, token refresh і маршрутизації запитів по інсталяціях.

## Прийняте рішення

Обрано **дедуплікацію per-repo** як обов'язкову базову стратегію для цільового навантаження.

Scanner має групувати активні підписки за унікальним репозиторієм, виконувати один запит до GitHub API на репозиторій, а після виявлення нового релізу створювати сповіщення для всіх підписників цього репозиторію.

Додатково Scanner має бути **rate-budget aware**: використовувати authenticated requests, контролювати `x-ratelimit-remaining` / `x-ratelimit-reset`, обмежувати concurrency і не продовжувати polling, якщо це веде до rate-limit помилок.

**Conditional requests через ETag / If-Modified-Since** прийнято використовувати як оптимізацію polling, а не як окрему архітектурну гарантію. Вони корисні для типового release polling, де більшість перевірок не знаходить нового релізу і отримує `304 Not Modified`. Але вони не повинні бути єдиним аргументом, що система витримає навантаження.

Для release polling не використовується TTL-кеш повних відповідей GitHub API як джерело істини, бо stale cache може приховати новий реліз і порушити 10-хвилинний polling objective. Scanner має зберігати cache metadata (`ETag`, `Last-Modified`, rate-limit headers) разом зі станом репозиторію і виконувати conditional request до GitHub при кожній плановій перевірці. Короткий cache для subscribe-time repository validation допустимий, але він не замінює polling.

При оцінці з `system-design.md` приблизно 500-700 унікальних репозиторіїв з інтервалом 10 хвилин дають близько 3 000-4 200 HTTP-викликів на годину. Це вкладається у стандартний authenticated primary limit 5 000 запитів на годину на один token і залишає помірний запас для валідації репозиторіїв, retry та secondary rate limits. Для дотримання 10-хвилинної затримки потрібна комбінація: per-repo дедуплікація, conditional requests, rate-budget aware scheduler і моніторинг використання GitHub API budget.

Якщо кількість унікальних репозиторіїв зростає за межі цього профілю і система перестає вкладатися в rate budget, потрібно змінювати одну з базових умов: збільшувати доступний GitHub API budget, зменшувати частоту polling для частини репозиторіїв, або переходити на webhook-модель для репозиторіїв, де доступні права адміністрування.

GitHub Webhooks не обираються як базове рішення, бо вони вимагають публічного endpoint-а та прав адміністрування репозиторіїв, що погано сумісно з моделлю, де користувач підписується на довільний `owner/repo`.

GitHub App не використовується в поточному рішенні, але приймається як запланований майбутній напрям розвитку. Це хороший вибір для репозиторіїв або організацій, де користувачі готові встановити App, бо installation access token може дати до 12 500 запитів на годину, а для GitHub Enterprise Cloud organization - 15 000 запитів на годину. Такі ліміти потрібні для подальшого росту кількості унікальних репозиторіїв, але не є обов'язковими для поточного профілю навантаження.

## Наслідки

### Позитивні

- Кількість запитів до GitHub API залежить від кількості унікальних репозиторіїв, а не від кількості підписників.
- Архітектура не потребує публічного webhook endpoint-а для довільних репозиторіїв.
- Логіка добре масштабується для сценаріїв, де багато користувачів підписані на однакові репозиторії.
- Rate-budget aware scheduler робить поведінку Scanner-а контрольованою при наближенні до GitHub API limit.
- Conditional requests зменшують primary rate limit usage для репозиторіїв без змін.
- GitHub App є зрозумілим шляхом збільшення API budget для репозиторіїв, де можлива інсталяція App.

### Негативні

- Система залишається polling-based, тому релізи не доставляються миттєво.
- Потрібно зберігати стан останнього побаченого релізу для кожного репозиторію.
- Conditional requests не зменшують кількість HTTP-викликів і не гарантують економію rate limit при `200 OK` відповідях.
- Якщо кількість унікальних репозиторіїв суттєво перевищить 700, 10-хвилинний polling може вичерпати стандартний GitHub API limit без додаткового API budget або webhook-стратегії.
- Якщо rate budget недостатній, доведеться переглядати SLA polling, спосіб автентифікації або webhook-стратегію.
- GitHub App не покриває довільні репозиторії автоматично: для доступу через installation token потрібна встановлена App.
