# ADR-005: Стратегія retry для email-провайдера

**Статус:** Прийнято

**Дата:** 2026-05-10

**Автор:** Іван Дутов

## Контекст

Sender надсилає email-сповіщення через зовнішнього провайдера, наприклад Resend. Email-провайдер може тимчасово повертати помилки: 5xx, timeout, rate limit або мережеві збої.

Якщо система просто логує помилку і переходить до наступного повідомлення, вимога **at-least-once delivery** порушується. Тимчасовий збій провайдера не має призводити до остаточної втрати сповіщення.

Поточна реалізація використовує in-memory buffered channel між Scanner та Sender-ом для release notification emails. Confirmation emails надсилаються API Server-ом напряму через Email Provider Integration. Персистентної черги, таблиці outbox або durable статусів доставки наразі немає. Це означає, що retry-стратегія не може покладатися на `next_attempt_at`, `attempt_count` або dead-letter статуси без окремої зміни storage-моделі, описаної в ADR-002.

Потрібно визначити retry-стратегію для цільового навантаження і вимоги at-least-once delivery з `system-design.md`.

Через обмеження часу поточна реалізація не включає durable retry scheduler. Current-state retry може бути тільки локальним inline retry у межах живого процесу. Target-state retry має спиратися на persistent email outbox, щоб переживати рестарти, довгі outage-и провайдера і rate-limit сценарії.

## Розглянуті варіанти

### 1. Drop on error

При помилці від email-провайдера система логує її і більше не намагається надіслати повідомлення.

- **Плюси:** найпростіша реалізація; немає ризику накопичення черги; легко зрозуміти поведінку.
- **Мінуси:** повідомлення втрачаються при тимчасових збоях; порушується at-least-once; користувач може не отримати важливе сповіщення.

### 2. Retry з backoff у межах одного циклу Sender

Sender повторює спробу надсилання кілька разів із паузою, яка збільшується після кожної невдалої спроби.

- **Плюси:** захищає від коротких збоїв провайдера; не потребує персистентної черги; просто додати до поточного Sender-а.
- **Мінуси:** не переживає падіння процесу між спробами; блокує обробку повідомлення на час retry; не підходить для довгих outage-ів провайдера.

### 3. Retry через повторну постановку в чергу

При помилці повідомлення повертається в чергу з delay і буде оброблене пізніше.

- **Плюси:** краще працює при довших збоях; не блокує Sender на одному повідомленні; добре масштабується разом із персистентною чергою.
- **Мінуси:** потребує durable queue або таблиці pending-сповіщень; складніше керувати кількістю спроб, backoff і конкурентною обробкою.

### 4. Dead letter queue

Після вичерпання кількості спроб повідомлення переміщується в окрему чергу або таблицю для ручного аналізу, повторного запуску чи алертингу.

- **Плюси:** повідомлення не зникають безслідно; зручно аналізувати постійні помилки; дає операційний контроль у production.
- **Мінуси:** потребує додаткових таблиць або статусів; потрібно визначити процес ручної обробки й алертингу.

## Прийняте рішення

Для поточної реалізації прийнято **best-effort delivery без durable retry**.

Поточний Sender не повертає повідомлення в чергу після помилки доставки. Якщо batch не вдалося доставити в межах поточного процесу, помилка логується, метрики доставки мають це відобразити, але повідомлення не повертається в durable queue, бо такої черги немає. Це прийнятий current-state компроміс через брак часу на реалізацію outbox-схеми, міграцій, backoff scheduler-а і конкурентного locking protocol.

Короткий inline retry у межах одного flush-циклу може бути доданий як локальне покращення, якщо він не блокує pipeline надовго і не створює retry storm при rate limit-ах провайдера. Але такий retry все одно не переживає рестарт процесу і не замінює durable delivery.

Для target-state production-архітектури заплановано перейти на **retry через повторну постановку в персистентну чергу** з backoff. У цьому варіанті Sender має оновлювати `next_attempt_at`, `attempt_count` і статус повідомлення в durable storage, а після вичерпання кількості спроб переводити повідомлення в **dead letter queue** або окремий фінальний статус для ручного аналізу, повторного запуску чи алертингу. Детальна специфікація наведена в `docs/email-outbox-spec.md`.

## Наслідки

### Позитивні

- Поточна реалізація явно не обіцяє durable retry і не створює прихованої гарантії, яку неможливо виконати без persistent outbox.
- Якщо буде додано короткий inline retry, він може зменшити втрати при дуже коротких збоях провайдера, але це залишається локальним покращенням, а не delivery guarantee.
- Поточна реалізація залишається простою і не потребує додаткових таблиць, locking protocol або scheduler-а для delayed jobs.
- Sender не накопичує необмежену кількість повідомлень при довгому outage-і email-провайдера.
- Запланований перехід на durable retry узгоджується з ADR-002 і дає зрозумілий шлях до at-least-once delivery.

### Негативні

- Поточний retry не переживає рестарт процесу і не виконує повну at-least-once гарантію.
- Повідомлення можуть бути втрачені при довгому outage-і email-провайдера, rate limit-і або падінні процесу після взяття batch-а в обробку.
- Для production-рівня потрібні додаткові поля або таблиці для статусу доставки, часу наступної спроби і кількості спроб.
- Після переходу на durable retry потрібно класифікувати помилки на тимчасові та постійні, а також додати алерти або регулярний огляд dead letter queue.
