# ADR-003: Механізм підтвердження підписки

**Статус:** Прийнято

**Дата:** 2026-05-10

**Автор:** Іван Дутов

## Контекст

Система використовує double opt-in: користувач створює підписку, отримує email і має підтвердити, що він справді хоче отримувати сповіщення.

До системи була висунута зовнішня вимога щодо API flow: підтвердження підписки має виконуватися через `GET /api/confirm/{token}`. Це частина заданої структури endpoint-ів системи разом із іншими публічними операціями підписки, перегляду та відписки.

Користувач має отримати лист і перейти за confirmation link без створення акаунта, введення пароля або проходження додаткових кроків.

## Розглянуті варіанти

### 1. GET підтверджує одразу

Посилання з email веде на endpoint, який одразу активує підписку.

- **Плюси:** відповідає зовнішній вимозі до API flow; найпростіший користувацький сценарій; не потребує окремої сторінки або форми; добре працює в email-клієнтах.
- **Мінуси:** GET-запит змінює стан системи; email-сканер або security proxy може перейти за посиланням автоматично.

### 2. GET відкриває сторінку, POST підтверджує

GET-посилання з email відкриває сторінку підтвердження. На сторінці є кнопка, яка виконує POST-запит.

- **Плюси:** GET не змінює стан системи; підписка активується лише після явної дії користувача.
- **Мінуси:** додає один крок у сценарій; не відповідає заданому API contract без додаткового endpoint-а.

## Прийняте рішення

Обрано варіант **GET підтверджує одразу**.

Confirmation email містить одноразове посилання. Перехід за цим посиланням викликає `GET /api/confirm/{token}` і підтверджує підписку без додаткових дій з боку користувача.

Рішення передбачає:

- валідує формат токена;
- шукає subscription за `confirm_token`;
- блокує рядок через `SELECT ... FOR UPDATE`;
- встановлює `confirmed = TRUE`;
- повертає `NotFound`, якщо токен не знайдено.

Expiry timestamp для confirmation token не вводиться. Тому прострочення токена не перевіряється.

## Наслідки

### Позитивні

- Найменший friction для користувача: один клік із confirmation email.
- Flow прямо відповідає зовнішньо заданому API contract системи.
- Не потрібна окрема confirmation page з формою або кнопкою.
- Повторне підтвердження того самого token є безпечним для стану subscription, бо `confirmed` лишається `TRUE`.

### Негативні

- GET-запит змінює стан системи, що відходить від strict HTTP-семантики.
- Email-клієнти, антивіруси або корпоративні security proxy можуть автоматично перейти за посиланням і підтвердити підписку без явного кліку користувача.
- Токени не мають expiration у схемі цього рішення.
