# ADR-004: Механізм підтвердження підписки

**Статус:** Прийнято

**Дата:** 2026-05-10

**Автор:** Іван Дутов

## Контекст

Система використовує double opt-in: користувач створює підписку, отримує email і має підтвердити, що він справді хоче отримувати сповіщення.

До системи була висунута зовнішня вимога щодо API flow: підтвердження підписки має виконуватися через `GET /api/confirm/{token}`. Це частина заданої структури endpoint-ів системи разом із іншими публічними операціями підписки, перегляду та відписки.

Користувач має отримати лист і перейти за confirmation link без створення акаунта, введення пароля або проходження додаткових кроків.

Потрібно обрати механізм, який:

- реалізує double opt-in;
- відповідає зовнішньо заданому API contract;
- не потребує логіну;
- мінімізує кількість дій для користувача;
- легко працює з email-клієнтів;
- достатній для поточного рівня ризику системи.

## Розглянуті варіанти

### 1. GET підтверджує одразу

Посилання з email веде на endpoint, який одразу активує підписку.

- **Плюси:** відповідає зовнішній вимозі до API flow; найпростіший користувацький сценарій; не потребує окремої сторінки або форми; добре працює в email-клієнтах; легко пояснити користувачу.
- **Мінуси:** GET-запит змінює стан системи; email-сканер або security proxy може перейти за посиланням автоматично; можливі випадкові підтвердження.

### 2. GET відкриває сторінку, POST підтверджує

GET-посилання з email відкриває сторінку підтвердження. На сторінці є кнопка "Підтвердити підписку", яка виконує POST-запит.

- **Плюси:** GET не змінює стан системи; підписка активується лише після явної дії користувача; email-сканери зазвичай не виконують POST; відповідає стандартній практиці double opt-in.
- **Мінуси:** додає один крок у сценарій; потрібно реалізувати сторінку підтвердження; потрібно обробити повторне або прострочене використання токена.

### 3. GET відкриває сторінку з додатковою взаємодією

Після переходу користувач має виконати додаткову дію: пройти CAPTCHA, поставити чекбокс або іншим способом підтвердити, що це не автоматичний перехід.

- **Плюси:** максимальний захист від автоматичних клієнтів; корисно при високому ризику abuse.
- **Мінуси:** додає friction; погіршує UX; надмірно для звичайного підтвердження email-підписки.

## Прийняте рішення

Обрано варіант **GET підтверджує одразу**.

Confirmation email містить одноразове посилання. Перехід за цим посиланням підтверджує підписку без додаткових дій з боку користувача.

Це рішення прийняте через зовнішню вимогу до API contract системи: confirmation endpoint має бути саме GET endpoint-ом з токеном у path. Для системи без акаунтів і паролів такий flow також знижує friction і підвищує ймовірність завершення підписки.

Endpoint підтвердження має коректно обробляти:

- валідний токен;
- прострочений токен;
- уже використаний токен;
- неіснуючий або пошкоджений токен.

## Наслідки

### Позитивні

- Найменший friction для користувача: один клік із confirmation email.
- Flow прямо відповідає зовнішньо заданому API contract системи.
- Не потрібна окрема confirmation page з формою або кнопкою.
- Простий no-login сценарій краще підходить для email-first сервісу.

### Негативні

- GET-запит змінює стан системи, що відходить від strict HTTP-семантики.
- Email-клієнти, антивіруси або корпоративні security proxy можуть автоматично перейти за посиланням і підтвердити підписку без явного кліку користувача.
- Потрібно мати достатньо стійкі, непередбачувані та одноразові токени.
- Якщо ризик автоматичного prefetch стане суттєвим, потрібно перейти до варіанта GET page + POST confirm.
